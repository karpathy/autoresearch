    - program.md (understand the crash recovery protocol)
    - results.tsv (current state -- note the row count before crash test)
  </read_first>
  <action>
    Test crash recovery by intentionally triggering an OOM error inside the training path.

    Steps:
    1. Record current state:
       - `wc -l results.tsv` to know the row count before crash test
       - `git log --oneline -1` to know the current HEAD commit
    2. Insert OOM trigger into train.py. Add this line near the top of the training function (MUST be inside the code path that Phase 2's crash handler wraps):
       ```python
       _oom_trigger = torch.zeros(24_000, 1024, 1024, device="cuda", dtype=torch.float32)  # ~24GB, exceeds RTX 4090
       ```
       IMPORTANT: Read the crash recovery code first to identify exactly which function/block is wrapped in try/except for torch.cuda.OutOfMemoryError. The trigger MUST be placed inside that block.
    3. Commit the OOM trigger: `git add train.py && git commit -m "test: intentional OOM trigger for crash recovery validation"`
    4. Run the experiment: `python train.py > run.log 2>&1`
    5. Verify crash was CAUGHT (not a hard process crash):
       - The Python process should exit cleanly (exit code 0) or the loop should continue
       - `grep "crash" results.tsv` should show a new row with status "crash"
       - results.tsv row count should have increased by 1
    6. Verify git reset was performed:
       - `git log --oneline -3` should show the OOM commit was reverted
       - `git diff` should show clean working tree
       - train.py should NOT contain the `_oom_trigger` line anymore
    7. Verify the system can continue:
       - If the experiment loop is running, it should proceed to the next iteration
       - If single-shot, verify the process exited cleanly and could be restarted

    If the OOM is NOT caught (process crashes hard): this indicates Phase 2's crash handler does not cover the code path where the trigger was placed. Move the trigger to inside the wrapped block and retry.
  </action>
  <verify>
    <automated>grep -c "crash" results.tsv | grep -q "[1-9]" && ! grep -q "_oom_trigger" train.py && echo "PASS: crash logged and OOM trigger removed" || echo "FAIL: crash not logged or OOM trigger still in train.py"</automated>
  </verify>

    - The crash row in results.tsv has the correct commit hash matching the OOM experiment commit
    - Git log shows the OOM commit was reverted: the current HEAD is the pre-OOM commit
  </acceptance_criteria>
